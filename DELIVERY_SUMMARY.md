## The Core Innovation

**Problem:** Podcasts get repetitive. Hosts become NPCs wearing microphones.

**Solution:** Treat character evolution like infrastructure.

- **Observe** patterns in transcripts (automated)
- **Classify** them (loop, ritual, callback)
- **Patch** intentionally with rationale (human decision)
- **Version** in git with explanations (audit trail)
- **Share** occasionally with listeners (transparency)

**Result:** A show that learns visibly, stays alive intentionally, and documents the process.

## Deliverable #3: Post-Episode Maintenance System

### Cadence

**Monthly (not weekly):**
1. Post-episode analysis optional (automated seed for decisions)
2. Luis editorial review (15 min, ~1 decision per episode)
3. SOUL.md patches approved and committed to git
4. VERA Patch Notes appear every 4–8 episodes (only when real changes)

### VERA Patch Notes Format

1-2 minute segment appearing occasionally:

```
Deprecated:
  - "phrase" (reason, replacement)

Promoted:
  - "new phrase or behavior" (why it works)

Monitor:
  - "ritual name" (frequency, saturation concern)

Growth Edge:
  - Hal/Ada: [what we're seeding] (success metric)

NPC Risk: [status]
```

### Governance Model

- **Default:** No patch (only clear wins)
- **Authority:** Luis reviews, hosts have voice
- **Transparency:** Every change is a git commit with explanation
- **Scalability:** 1-3 decisions per episode, manageable overhead
- **Sustainability:** System defaults to "no change" not constant iteration

### Boundaries (What NOT To Do)

❌ Weekly mandatory patch notes (too much)  
❌ Publish full SOUL.md publicly (shows only what matters)  
❌ Recursive SOUL.md (no VERA's VERA; Hal/Ada/VERA only)  
❌ Hard bans (soft observation; deprecated ≠ forbidden)  
❌ Metrics obsession (infrastructure, not performance system)
